# 管理端申请详情页打印申请单设计

## 背景

当前申请详情页位于 `app/admin/(auth)/applications/[id]/page.tsx`，包含管理员审核所需的完整信息与附件预览。现在需要在该详情页增加“打印申请单”能力，用于输出纸质表单。

这次打印有两个明确约束：

1. 打印动作必须在当前详情页内直接触发浏览器打印。
2. 打印内容必须是一套独立于当前详情页的新区域，不能沿用当前卡片布局。

## 目标

在管理员查看单个申请详情时，能够点击“打印申请单”按钮，直接调起浏览器打印，并输出一张适合 `A4 纵向` 的申请单。纸质申请单需要包含：

- 页头标题
- 申请编号
- 打印时间
- 基本信息
- 监护人信息
- 学校与地址
- 指向 `/application/pending/[id]` 的查询结果二维码
- 签字栏

打印时必须隐藏当前页面的非打印内容，包括导航、审批操作、附件区和原有详情布局。

## 非目标

本次不包含以下内容：

- 新增独立打印路由
- 新开浏览器窗口或弹窗进行打印
- 打印附件图片
- 修改公众侧 `/application/pending/[id]` 页面逻辑
- 重构 `getApplicationById` 数据读取方式

## 当前上下文

### 页面与数据

- 管理端详情页通过 `getApplicationById(id)` 在服务端读取申请详情。
- 公众侧待审核查询页位于 `app/application/pending/[id]/page.tsx`。
- 当前详情页已经展示了本次打印所需的大部分字段：
  - 基本信息：姓名、性别、身份证号、学籍号、户籍类型、学期
  - 监护人信息：监护人1、监护人2姓名与电话
  - 学校与地址：当前学校、当前年级、目标年级、分配学校、户籍地址、居住地址

### 技术边界

根据当前项目的 Next.js App Router 约束：

- `page.tsx` 默认应保持为 Server Component，用于服务端读取数据。
- 浏览器事件和 `window.print()` 必须放在单独的 Client Component 中。
- 打印模板可以由 Server Component 渲染，只要不依赖浏览器 API。

## 方案对比

### 方案 A：同页独立打印模板加打印样式

做法：

- 在当前详情页中新增一个打印按钮组件。
- 在当前详情页中新增一个独立的打印模板区域。
- 屏幕态显示原页面，隐藏打印模板。
- 打印态隐藏原页面，只显示打印模板。

优点：

- 完全符合“当前页直接打印”的要求。
- 可复用当前页已获取的数据，不新增路由与请求。
- 与现有审核逻辑耦合很低，只新增打印相关组件。

缺点：

- 页面 DOM 中会并存屏幕布局和打印布局两套结构。

### 方案 B：新增独立打印页

做法：

- 新增打印路由，点击后跳转到纯打印页面，再调用打印。

优点：

- 打印 DOM 最干净。

缺点：

- 不符合“在当前详情页内直接调起打印”的要求。
- 需要新增路由与跳转状态管理。

### 方案 C：新窗口写入 HTML 后打印

做法：

- 点击按钮后打开新窗口，运行时拼接一份 HTML 模板并打印。

优点：

- 可以完全隔离当前页面。

缺点：

- 数据与样式维护成本高。
- 打印兼容性和可维护性最差。

## 结论

采用方案 A。

这是最符合当前需求的实现路径：在同一个详情页内触发打印，但打印输出内容完全来自一套新写的打印模板，而不是当前详情页本身。

## 目标实现结构

### 1. 服务端详情页继续负责取数

文件：`app/admin/(auth)/applications/[id]/page.tsx`

职责：

- 继续通过 `getApplicationById(id)` 获取申请详情。
- 继续渲染现有管理员详情页。
- 额外挂载打印按钮组件。
- 额外挂载打印模板组件。

该文件不直接调用 `window.print()`，保持 Server Component。

### 2. 新增客户端打印按钮组件

建议文件：`app/admin/(auth)/applications/_components/print-application-button.tsx`

职责：

- 使用 `"use client"`。
- 渲染“打印申请单”按钮。
- 点击后调用 `window.print()`。
- 按钮仅在屏幕态显示，打印态隐藏。

不负责：

- 读取数据
- 生成打印内容
- 管理打印状态

### 3. 新增打印模板组件

建议文件：`app/admin/(auth)/applications/_components/application-print-sheet.tsx`

职责：

- 接收申请详情数据作为 props。
- 渲染独立的 A4 打印布局。
- 默认屏幕态隐藏，打印态显示。
- 不依赖当前详情页的卡片布局。

## 打印模板内容设计

打印单采用 `A4 纵向`，内容顺序如下。

### 1. 页头

包含：

- 主标题：`转学申请单`
- 学期名称：来自 `app.semester.name`
- 申请编号：`app.id`
- 打印时间：打印触发时的当前时间

说明：

- 打印时间属于浏览器侧信息，应由客户端在打印模板中生成。
- 如果实现上希望保持模板本身为纯展示组件，可以由打印按钮组件或一个更薄的客户端壳层将格式化后的打印时间作为字符串传入。

### 2. 基本信息

包含：

- 学生姓名：`app.name`
- 性别：`app.gender`
- 身份证号：`app.idCard`
- 学籍号：`app.studentId`
- 户籍类型：`app.residencyType`
- 申请学期：`app.semester.name`

要求：

- 采用正式表单式布局，不沿用屏幕态卡片分组。
- 户籍类型显示中文文案。

### 3. 监护人信息

包含：

- 监护人1姓名：`app.guardian1Name`
- 监护人1电话：`app.guardian1Phone`
- 监护人2姓名：`app.guardian2Name`
- 监护人2电话：`app.guardian2Phone`

要求：

- 第二监护人不存在时，固定显示 `无`，不留不可判断的空白。

### 4. 学校与地址

包含：

- 当前学校：`app.currentSchool`
- 当前年级：`app.currentGrade`
- 目标年级：`app.targetGrade`
- 分配学校：`app.targetSchool`
- 户籍地址：`app.hukouAddress`
- 居住地址：`app.livingAddress`

要求：

- 当前年级与目标年级拆成两个独立字段，避免纸面表达过于 UI 化。
- 分配学校为空时显示 `尚未分配`。

### 5. 查询结果二维码

二维码目标固定为：

- `/application/pending/[id]`

无论申请状态如何，二维码均统一跳转到该地址。

二维码区需同时包含：

- 二维码图形
- 提示文字，例如“扫码查看申请处理进度”
- 明文查询路径或完整链接，作为扫码失败时的兜底信息

说明：

- 公众页本身如果遇到已通过或已驳回状态，会继续沿用现有逻辑跳转到对应结果页。
- 打印侧无需根据状态动态改变二维码目标。

### 6. 签字栏

打印单底部预留：

- 监护人签字
- 审核人签字
- 备注
- 日期

要求：

- 作为纸质填写区域展示，无需与数据库绑定。

## 样式与打印行为

### 屏幕态

- 保持现有详情页外观不变。
- 打印模板区域隐藏。
- 打印按钮可见。

### 打印态

- 隐藏现有详情页的所有非打印内容。
- 仅显示打印模板。
- 打印按钮、导航按钮、审批面板、附件区全部隐藏。

### 版式要求

- 页面宽度按 A4 纵向纸张控制。
- 优先使用项目现有 Tailwind 工具类与 `print:` 变体完成样式切换。
- 表单内容应稳定、规整、适合黑白打印。
- 避免依赖深色背景、大面积阴影、动画或交互式视觉元素。

### 分页要求

- 本次打印内容原则上控制在单页内。
- 不打印附件图像，避免内容过长。
- 关键区域如二维码区和签字栏应尽量避免被分页拆开。

## 数据与字段兜底策略

打印模板中对可空字段采用明确文案兜底：

- `guardian2Name` / `guardian2Phone` 为空时显示 `无`
- `targetSchool` 为空时显示 `尚未分配`

若二维码组件生成异常：

- 打印模板仍显示文字链接
- 不应导致整个打印页面崩溃

若申请不存在：

- 继续沿用当前页面 `notFound()` 行为

## 二维码实现策略

二维码内容需要是可扫码访问的 URL。

优先策略：

- 使用当前站点的绝对地址拼接 `/application/pending/${app.id}`

可接受策略：

- 若当前实现难以在服务端稳定获得站点 origin，则由客户端在渲染时使用 `window.location.origin` 生成完整 URL

兜底要求：

- 同时打印文字路径 `/application/pending/${app.id}` 或完整链接

实现上允许引入轻量二维码组件库，只要不会破坏当前页面结构与测试稳定性。

## 风险与控制

### 风险 1：服务端页面与客户端打印时间/二维码 origin 协调

控制：

- 将浏览器侧信息收敛到极薄的客户端边界中。
- 服务端仍只负责业务数据。

### 风险 2：打印样式误伤屏幕态布局

控制：

- 使用清晰的打印容器类名和 `print:` 条件样式。
- 不对现有页面结构做大范围重写。

### 风险 3：二维码库引入后影响测试或打包

控制：

- 优先选择简单、无副作用、可在 React 中直接渲染的轻量实现。
- 测试只断言关键输出，不依赖复杂图形细节。
