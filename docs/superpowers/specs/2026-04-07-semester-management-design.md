# 学期管理页修复设计

## 背景

当前仓库已经引入 Prisma，学期相关服务端操作集中在 `app/actions/semester.ts`，学期切换入口位于 `components/admin/semester-switcher.tsx`。但 `/admin/semesters` 页面仍然直接查询 Prisma，且只具备部分新增和删除能力，编辑、启停和切换器联动尚未完成。

本次工作仅聚焦 `app/admin/(auth)/semesters/page.tsx` 对应的学期管理闭环，不扩展到其他业务页面。

## 目标

1. 完成学期列表、新增、编辑、删除、启用/停用能力。
2. 让 `/admin/semesters` 页面只依赖 `app/actions/semester.ts`，不在页面内直接调用 Prisma。
3. 新增和编辑沿用弹窗交互，不改为独立详情页。
4. 规范学期新增方式，减少自由输入带来的命名和日期错误。
5. 保证学期变更后，`components/admin/semester-switcher.tsx` 能正确反映最新列表和当前选中项。

## 非目标

1. 不修改 Prisma 的 `Semester` 数据模型结构。
2. 不实现跨页面的“当前学期”持久化选择机制。
3. 不调整学期切换器在侧边栏中的整体视觉结构，只修正其数据联动和显示逻辑。
4. 不扩展到“申请新增时如何校验当前学期可用”的业务实现，只在停用提示文案中明确风险。

## 现状问题

1. `app/admin/(auth)/semesters/page.tsx` 直接查询 Prisma，绕过了 `app/actions/semester.ts`。
2. `app/actions/semester.ts` 只有读取、新增、删除，缺少编辑和启停。
3. `components/admin/semester/create-dialog.tsx` 直接暴露名称输入，容易产生不标准命名。
4. `lib/validations/semester.ts` 未校验结束日期晚于开始日期。
5. `components/admin/semester-switcher.tsx` 将初始学期塞入本地 state 后不再随 `semesters` 变化自动纠正，新增、删除或停用后可能显示过期对象。

## 用户流程

### 学期新增

1. 管理员进入 `/admin/semesters`。
2. 点击“新增学期”打开弹窗。
3. 依次选择年度、学期季度，并确认系统自动生成的日期范围。
4. 如有需要，在允许范围内微调开始日期和结束日期，并选择是否启用。
5. 提交后关闭弹窗，列表刷新，侧边栏切换器同步刷新。

### 学期编辑

1. 管理员在列表中点击“编辑”。
2. 弹窗预填当前学期信息，并反推出年度与季度。
3. 管理员调整日期或启用状态后提交。
4. 列表和侧边栏学期切换器同步刷新。

### 学期启停

1. 管理员在列表中点击“停用”或“启用”。
2. 停用时显示确认弹窗，明确提示“关闭会导致当前学期无法新增任何申请”。
3. 确认后更新 `isActive`，列表和侧边栏同步刷新。
4. 已停用学期仍保留在列表中，并允许重新启用。

### 学期删除

1. 管理员在列表中点击删除。
2. 删除确认弹窗显示学期名称和不可撤销说明。
3. 删除成功后刷新列表和侧边栏切换器。
4. 如果当前切换器显示的学期被删除，自动回退到“当前日期命中的学期”；若没有命中项，则回退到排序后的第一项。

## 数据设计

### 复用现有 Prisma 模型

继续使用现有 `Semester` 字段：

- `id`
- `name`
- `startDate`
- `endDate`
- `isCurrent`
- `isActive`
- `createdAt`
- `updatedAt`

其中：

- `isCurrent` 不在本次页面中暴露编辑能力。
- `isActive` 需要在新增和编辑中可控，也需要支持列表中的快捷启停。

### 表单输入模型

为了便于用户新增标准学期，弹窗中的交互模型与数据库模型解耦。用户看到的字段为：

- `year`
- `term`
- `startDate`
- `endDate`
- `isActive`

提交前再转换成数据库需要的值：

- `name = ${year}年${term}`
- `startDate = 用户最终选择日期`
- `endDate = 用户最终选择日期`
- `isActive = 用户选择值`

### 年度和季度规则

- 年度候选按“当前年份前后各 5 年”生成。
- 以当前环境日期 `2026-04-07` 为例，候选区间明确为 `2021` 到 `2031`。
- 季度候选固定为 `春季`、`秋季`。
- 名称固定自动生成，不允许手动修改：
  - `2026年春季`
  - `2026年秋季`

### 默认日期模板

#### 春季

- 名称示例：`2026年春季`
- 默认开始日期：`2025-09-01`
- 默认结束日期：`2026-03-01`
- 允许用户选择的日期范围限定在 `2025-09-01` 到 `2026-03-01` 之间

#### 秋季

- 名称示例：`2026年秋季`
- 默认开始日期：`2026-03-01`
- 默认结束日期：`2026-09-01`
- 允许用户选择的日期范围限定在 `2026-03-01` 到 `2026-09-01` 之间

### 编辑时的反推规则

编辑已有学期时，需要根据已有记录反推出年度和季度，用于初始化统一表单：

- 若 `startDate` 与 `endDate` 跨自然年，则视为 `春季`，年度取 `endDate` 的年份。
- 若 `startDate` 与 `endDate` 在同一年，则视为 `秋季`，年度取 `startDate` 的年份。
- 若现有日期不完全等于模板默认值，仍允许编辑，但只要在该季度合法区间内即可。

## 页面设计

### 页面数据来源

`app/admin/(auth)/semesters/page.tsx` 继续作为 Server Component，仅负责：

1. 调用 `app/actions/semester.ts` 的读取函数获取学期列表。
2. 渲染页面标题、列表和弹窗入口。
3. 将单条记录传给编辑、删除、启停组件。

页面内不再直接访问 Prisma。

### 列表结构

列表展示全部学期，包括已停用项。字段如下：

1. 学期名称
2. 起止日期
3. 时间状态
4. 启用状态
5. 操作

#### 时间状态

按当前日期与 `startDate`、`endDate` 的关系计算：

- `未开始`
- `进行中`
- `已结束`

#### 启用状态

按 `isActive` 显示：

- `已启用`
- `已停用`

时间状态和启用状态必须分开显示，避免把“日期进行中”与“业务可用”混为一谈。

### 操作区

每行提供以下操作：

1. `编辑`
2. `停用` 或 `启用`
3. `删除`

其中：

- `编辑` 打开统一学期表单弹窗。
- `停用` 需要确认弹窗并提示风险。
- `启用` 可以使用轻量确认或与停用一致的确认框。
- `删除` 继续使用确认弹窗，并统一返回成功或错误提示。

## 组件边界

### `components/admin/semester/semester-form-dialog.tsx`

新增一个统一表单弹窗组件，同时服务新增与编辑：

- `mode: "create" | "edit"`
- `semester?: Semester`
- `trigger?: ReactNode`

职责：

1. 负责年度、季度、日期、启用状态表单渲染。
2. 根据年度和季度自动生成名称。
3. 根据年度和季度计算可选日期范围与默认值。
4. 在新增模式调用 `createSemester`。
5. 在编辑模式调用 `updateSemester`。

### `components/admin/semester/create-dialog.tsx`

保留为轻薄包装层，内部直接复用统一表单弹窗的新增模式，避免页面引用关系大幅变化。

### `components/admin/semester/delete-button.tsx`

继续负责删除确认和删除动作调用，但对返回值和提示文案做统一收敛。

### `components/admin/semester/toggle-active-button.tsx`

新增启停按钮组件，负责：

1. 判断当前是启用还是停用动作。
2. 显示确认弹窗。
3. 执行 `toggleSemesterActive`。
4. 呈现成功或失败提示。

## 服务端动作设计

`app/actions/semester.ts` 统一承担学期数据访问和变更，提供：

1. `getSemesters()`
2. `createSemester(data)`
3. `updateSemester(id, data)`
4. `toggleSemesterActive(id, isActive)`
5. `deleteSemester(id)`

### 返回结构

所有变更动作统一返回以下形状之一：

- `{ success: true }`
- `{ error: string }`

### 服务端校验

所有新增和编辑动作都必须在服务端再次校验：

1. 名称不能为空，且应符合自动生成格式。
2. 开始日期和结束日期必须存在。
3. 结束日期不得早于开始日期。
4. 日期必须落在对应季度的合法区间内。
5. 名称唯一冲突时返回用户可理解的错误，例如“该学期已存在”。

### 刷新策略

新增、编辑、启停、删除成功后统一调用：

- `revalidatePath("/admin/semesters")`
- `revalidatePath("/admin")`

这样既能刷新学期管理页，也能刷新依赖学期列表的管理端布局和侧边栏切换器。

## 校验设计

`lib/validations/semester.ts` 需要从单纯的数据库输入校验，扩展为支持统一表单所需的验证逻辑。

建议拆分为两层：

1. 页面表单输入校验
   - 校验 `year`、`term`、`startDate`、`endDate`、`isActive`
2. 数据库存储输入校验
   - 校验 `name`、`startDate`、`endDate`、`isActive`

统一约束：

- `startDate <= endDate`
- 日期必须落在季度定义区间中
- `name` 由 `year + term` 计算得出，不接受任意自由文本

## 学期切换器联动

`components/admin/semester-switcher.tsx` 需要修正当前选中项的同步策略。

### 现有问题

组件当前只在首次渲染时基于 `semesters` 推导 `activeSemester`，随后把它保存到本地 state。若列表变更，本地 state 可能引用一个已删除或已过期的对象。

### 目标行为

1. 当 `semesters` 更新时，组件重新校正当前显示项。
2. 若当前显示项仍存在，则继续显示它。
3. 若当前显示项已不存在，则自动切到“当前日期命中的学期”。
4. 若当前日期无命中项，则回退到排序后的第一项。
5. 已停用学期仍显示在切换列表中，但需要在文案或标识上区分其状态。

### 范围说明

本次只修正切换器对列表变更的响应，不实现跨刷新持久化记住用户手动选择的学期。

## 错误处理

### 新增和编辑

- 若日期不在允许范围内，表单内展示明确错误。
- 若名称重复，弹窗保持打开并提示“该学期已存在”。
- 若服务端异常，提示“创建失败，请稍后再试”或“更新失败，请稍后再试”。

### 启停

- 若停用或启用失败，保留当前状态并提示错误。

### 删除

- 若数据库存在关联约束导致删除失败，返回“该学期存在关联数据，无法删除”之类的业务错误。

## 测试与验证策略

本次以页面闭环和关键联动为主，建议覆盖：

1. 学期管理页在无数据时展示空状态文案。
2. 学期管理页能展示全部学期，包括停用项。
3. 新增弹窗根据年度和季度自动生成名称。
4. 新增弹窗能根据季度给出正确的默认日期和合法日期范围。
5. 编辑弹窗能正确回填已有学期并反推季度。
6. 停用确认文案包含“关闭会导致当前学期无法新增任何申请”。
7. 删除当前切换器所显示的学期后，切换器自动回退到当前日期命中的学期。
8. `app/actions/semester.ts` 的新增、编辑、启停、删除都会触发目标路径刷新。

## 实施顺序

1. 调整 `lib/validations/semester.ts`，补足统一表单和日期区间校验。
2. 扩展 `app/actions/semester.ts`，补齐编辑与启停逻辑，并统一错误返回。
3. 抽出统一的学期表单弹窗组件，重构新增弹窗并接入编辑能力。
4. 新增启停按钮组件，复用现有删除组件。
5. 改造 `app/admin/(auth)/semesters/page.tsx`，改为通过 action 读取列表并补全操作列。
6. 修正 `components/admin/semester-switcher.tsx` 的联动逻辑。
7. 补充并执行相关测试。

## 风险与取舍

1. 现有 Prisma 模型包含 `isCurrent`，但本次不编辑它，页面状态需要明确区分“当前时间命中”和“业务启用”。
2. 统一表单需要将数据库记录反推为“年度 + 季度”，因此必须在文档中把跨年视为春季、同年视为秋季的规则写死，否则编辑回填会产生歧义。
3. 由于当前仓库工作区已存在大量未提交修改，本次实现必须严格限定在学期管理相关文件，避免误碰其他变更。
