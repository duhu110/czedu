# 学生转学管理端优化设计

## 背景

当前仓库已经具备学生端转学申请表单和教育局管理端骨架，但 `app/admin` 仍以通用英文 DEMO 为主，尚未形成围绕“转学申请审核”场景的完整闭环。本次工作目标是在现有骨架基础上，将管理端调整为可演示的中文后台：具备登录跳转、统计看板、申请列表、详情页、状态展示和基础导航体验。

## 目标

1. 将管理端首页统计区改造成“转学申请管理”语境，并填充合理的 mock 数据。
2. 将申请结果表格改造成真实的转学申请列表，数据字段参考学生端基础表单。
3. 新建学生申请详情页，用于查看单个申请的完整信息和审核状态。
4. 让管理端顶部导航在滚动时固定在页面顶部。
5. 完成管理端范围内的全局汉化，去除明显英文 DEMO 文案。
6. 建立一个纯 DEMO 的登录跳转闭环：`/` 或其他入口跳到 `/admin`，未登录自动进入 `/admin/login`，点击登录后返回 `/admin`。

## 非目标

1. 不接入真实后端、数据库或鉴权服务。
2. 不引入复杂的国际化框架。
3. 不实现真实导出文件，仅提供“模拟导出”交互。
4. 不重构学生端提交流程，仅复用其字段结构作为后台 mock 数据来源。

## 用户流程

### 管理员登录流程

1. 管理员从首页点击“进入管理端”或直接访问 `/admin`。
2. 如果未处于 DEMO 登录状态，页面自动跳转到 `/admin/login`。
3. 管理员点击登录按钮后，前端写入一个简单的 DEMO 登录标记。
4. 登录成功后跳转回 `/admin` 首页。
5. 之后访问 `/admin` 下的总览、申请列表、详情页均正常显示。
6. 若点击退出，则清除 DEMO 登录标记并返回 `/admin/login`。

### 审核查看流程

1. 管理员进入 `/admin` 查看统计卡片和申请趋势。
2. 管理员进入申请列表页查看分页数据、申请状态和关键信息。
3. 管理员点击某条记录的“查看详情”进入学生详情页。
4. 详情页显示学生基本信息、学校信息、联系方式、转学原因、审核状态、审核意见和补充资料要求。
5. 管理员点击“模拟导出”按钮获得前端提示，用于展示后续可扩展点。

## 信息架构

### 路由结构

- `/`
  - 首页，增加进入管理端的明显入口。
- `/admin`
  - 管理端入口。
  - 未登录时自动跳转到 `/admin/login`。
  - 已登录时展示统计总览页。
- `/admin/login`
  - DEMO 登录页。
- `/admin/users`
  - 申请列表页。
- `/admin/users/[id]`
  - 学生申请详情页。

### 导航结构

管理端侧边栏保留简洁结构，至少包含：

- 总览
- 申请管理
- 退出登录

顶部导航保留当前位置信息和用户区域，但统一为中文文案。

## 数据设计

### 申请数据模型

管理端新增一份专用 mock 数据，字段以学生端 `BasicInfo` 为核心，并补充后台审核字段：

- `id`
- `studentName`
- `studentId`
- `currentSchool`
- `currentGrade`
- `targetSchool`
- `targetGrade`
- `phone`
- `email`
- `reason`
- `status`
- `applyDate`
- `reviewDate`
- `reviewer`
- `progress`
- `resultSummary`
- `missingDocuments`
- `notes`

### 状态枚举

表格和详情页统一使用以下中文状态：

- `申请中`
- `已审核`
- `审核通过`
- `待补充资料`
- `已驳回`

其中：

- `申请中` 表示已提交，等待初审。
- `已审核` 表示教育局已审核但仍处于流程中，适合展示中间态。
- `审核通过` 表示转学申请通过。
- `待补充资料` 表示需补交材料。
- `已驳回` 表示申请未通过。

### 统计数据

首页统计区由申请数据聚合得出，至少包括：

- 申请总数
- 申请中数量
- 待补充资料数量
- 审核通过数量

趋势图使用最近一段时间的 mock 申请量数据，按日期展示申请量和办结量，不再沿用“desktop/mobile visitors”之类无关英文字段。

## 页面设计

### 1. 管理端首页

首页由三部分组成：

1. 统计卡片
   - 使用中文标题和说明。
   - 数值与转学申请业务相关。
2. 趋势图
   - 展示近 7 天、30 天、90 天申请趋势切换。
   - 坐标轴、tooltip 和说明文案全部汉化。
3. 申请列表预览
   - 复用申请列表表格组件或与其共享列定义。
   - 默认展示第一页。

### 2. 申请列表页

申请列表页需要满足：

- 使用管理端 mock 数据。
- 字段覆盖：学生姓名、学籍号、当前学校、目标学校、申请日期、状态、审核人、操作。
- 明确的状态徽标颜色区分。
- 分页能力可用。
- “查看详情”按钮跳转到详情页。
- “模拟导出”按钮放在列表工具栏，点击后提示导出成功或“已生成演示导出任务”。
- 空数据文案使用中文。

表格交互优先级：

1. 可读性优先。
2. 分页必须保留。
3. 拖拽排序、复杂编辑抽屉等与当前业务无关的 DEMO 能力可以删除或弱化，避免干扰审核主流程。

### 3. 学生详情页

详情页按内容分块展示：

1. 申请摘要
   - 当前状态
   - 申请编号
   - 申请日期
   - 审核人
2. 学生信息
   - 学生姓名
   - 学籍号
   - 当前年级
   - 目标年级
3. 学校信息
   - 当前学校
   - 目标学校
4. 联系方式
   - 手机号
   - 邮箱
5. 转学原因
   - 展示完整文本
6. 审核信息
   - 结果说明
   - 备注
   - 待补充资料列表

详情页应提供返回列表入口。

### 4. 登录页

登录页保留现有视觉骨架，但调整为中文 DEMO 登录场景：

- 标题、说明、副文案汉化。
- 表单字段改成适合管理端的账号/密码或账号/验证码形式。
- 登录按钮点击后写入简单登录态。
- 不做真实校验，只做最小的非空校验或直接允许演示登录。

### 5. 顶部导航固定

`AppNavbar` 需要在管理端内容滚动时固定在顶部：

- 使用 sticky 顶部吸附。
- 需要背景色与边框，避免下方内容滚动时重叠混乱。
- `layout.tsx` 需要调整容器结构，确保滚动容器和 sticky 生效。

## 汉化策略

本次不接入 i18n 框架，直接采用静态中文文案替换。

汉化范围：

- 管理端首页
- 管理端侧边栏
- 顶部导航
- 登录页
- 申请表格
- 分页文案
- 详情页
- 空状态、按钮、提示语

保留英文的唯一合理场景是邮箱地址、学籍号等用户数据字段值本身。

## 登录状态设计

采用纯 DEMO 登录态，不引入服务端鉴权。

### 存储方式

- 使用 `localStorage` 存储布尔标记，例如 `admin-demo-auth=true`。

### 访问控制

- 在管理端入口或受保护页面中判断该标记。
- 未登录则跳转到 `/admin/login`。
- 已登录访问 `/admin/login` 时可直接回到 `/admin`。

### 退出逻辑

- 点击退出按钮时清除本地标记。
- 然后跳转到 `/admin/login`。

此方案适合 DEMO，逻辑简单，能完整覆盖当前需要的跳转体验。

## 组件边界

建议保持以下责任划分：

- 管理端 mock 数据：独立放在 `lib` 或 `components/admin` 可复用位置。
- 首页统计组件：负责展示聚合后的申请统计。
- 首页趋势图组件：负责展示申请量趋势。
- 申请表格组件：负责列表、分页、导出按钮和行操作。
- 状态徽标渲染：独立抽出，保证首页、列表、详情状态展示一致。
- 详情页：只负责根据 `id` 读取数据并展示。
- DEMO 登录工具：负责读写登录标记。

这样可以避免页面文件中混入过多 mock、状态映射和导航判断逻辑。

## 错误处理

### 详情页不存在数据

- 若 `id` 对应的 mock 数据不存在，则进入 `notFound()` 或展示中文未找到页面。

### 未登录访问管理端

- 自动跳转到 `/admin/login`，不显示半成品内容。

### 已登录访问登录页

- 自动跳转到 `/admin`，避免重复登录。

## 测试与验证策略

本次以最小可验证闭环为主，验证点包括：

1. 访问 `/admin` 时，未登录自动跳到 `/admin/login`。
2. 在 `/admin/login` 点击登录后返回 `/admin`。
3. 首页统计卡片、趋势图、列表均显示中文 mock 数据。
4. 列表分页可切换。
5. 列表中的状态显示符合设计。
6. 点击查看详情能进入 `/admin/users/[id]`。
7. 详情页内容与列表行数据一致。
8. 管理端滚动时顶部导航固定。
9. 退出登录后重新访问 `/admin` 会回到 `/admin/login`。

## 实施顺序

1. 新增并整理管理端 mock 数据结构。
2. 完成 DEMO 登录态和 `/admin`、`/admin/login` 跳转闭环。
3. 改造侧边栏和顶部导航的中文化与导航逻辑。
4. 改造首页统计卡片和趋势图。
5. 改造申请列表表格、分页和模拟导出按钮。
6. 将用户详情动态路由调整为 `[id]` 并完成详情页展示。
7. 全量检查管理端英文遗留文案。

## 风险与取舍

1. 使用 `localStorage` 的登录态只能在客户端判断，因此需要谨慎放置跳转逻辑，避免闪烁。
2. 如果继续复用当前通用表格组件，可能会保留较多与业务无关的复杂交互；实现时应优先做减法。
3. 本次汉化为静态替换，不具备多语言切换能力，但对 DEMO 是合适取舍。
4. 当前仓库已存在较多未提交新增文件，本次修改需要避免误覆盖未纳入当前任务的内容。
